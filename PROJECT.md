## 1. プロダクト概要

- **プロダクト名**: DILEMMA（ジレンマ）— パーソナルトレーナー向け臨床意思決定シミュレータ
- **ビジョン**: 「仮想のクライアントを壊し、現実のクライアントを救え」
- **ポジショニング**: Pace-platform内の1機能（独立プロダクトではない）
- **ブランディング**: 「ゲーム」ではなく「パーソナルトレーナーのフライトシミュレータ」

### 解決する課題

パーソナルトレーナー（PT）教育における3つの構造的問題:

1. **座学と現場のギャップ**: 教科書知識だけでは、一般クライアント特有の複合リスク（生活習慣・既往歴・目的の多様性）が絡む複雑系の意思決定を学べない
2. **極端判断パターン**: 「休ませすぎ（クライアントの離反・退会）」と「追い込みすぎ（怪我・クレーム）」の二極化。中間の最適解を導く訓練機会が不足
3. **失敗コストの非対称性**: 現実では取り返しのつかない失敗（腰椎ヘルニア・膝損傷等）を経験してからしか学べない。仮想空間で安全に「取り返しのつかない失敗」を経験させる

### ターゲットユーザー

| セグメント | 詳細 | ITリテラシー |
|-----------|------|-------------|
| PT学生 | 専門学校・養成スクールのパーソナルトレーナー志望（在学中） | 中（スマホネイティブ、専門ツール経験少） |
| 駆け出しPT | 資格取得後1-3年、現場に出たばかりで不安を抱えるPT | 中〜高 |
| PT養成スクール | 教員・スクール運営者（教材として導入） | 中 |
| AT/S&C学生 | 大学のAT/S&C専攻（2〜4年次） | 中 |

### 収益モデル（LLM従量課金型）

**方針**: 月額定額ではなく、LLMの使用量に応じた従量課金。
ユーザーは「クレジット」を購入し、シミュレーション実行（LLMディベート・AAR生成）ごとにクレジットを消費する。

| プラン | 対象 | 価格 | 内容 |
|--------|------|------|------|
| Free | 学生/体験 | ¥0 | 月3シナリオ分の無料クレジット付与 |
| クレジット購入 | 全ユーザー | ¥100〜（最小購入単位）| 1シナリオ ≈ ¥50〜150相当（LLMトークン消費量による） |
| スクール・法人向け | PT養成スクール | 一括クレジット購入（ボリュームディスカウント） | クラス管理、学生進捗ダッシュボード、クレジット配布機能 |

### DILEMMA専用ラッパーAPI（新規設計）

```
【アーキテクチャ】
クライアント → DILEMMA ラッパーAPI → V7推論エンジン（pace-inference）

DILEMMAラッパーAPIの役割:
1. 「生活ノイズ」変数の受け取り（昨夜の飲酒、睡眠不足、デスクワーク時間等）
2. 生活ノイズ → V7が処理可能な数値（Z-scores等）への変換
3. 「今日のメニュー」+ 変換済み生活ノイズ → V7本体へリクエスト
4. V7結果の受け取り → 翌日の状態（state transition）を生成
5. セッション全体の状態遷移管理（Day 1→Day 2→...→Week End）
```

### エクササイズDB（PT向け再構築）

```
【方針】
- PACE（プロ用）のDBからベースを参照しつつ、PT特化で再構築
- 対象機材: ダンベル、バーベル、マシン、自重（一般的なパーソナルジム設備）
```

### Step 1: 処方と葛藤（Action & Dilemma）

- ユーザーがAT役（安全重視） or S&C役（成果重視）を選択
- 仮想クライアントのプロフィール（年齢、目的、生活習慣、既往歴、当日の生活ノイズ）を提示
- 「今日のメニューを決めてください」— エクササイズDB（RAG）から選択・カスタマイズ
- **対立AI**: 選択しなかった側の役割のAIが反論（ATを選んだらS&C AIが「もっと追い込まないと結果出ませんよ」と主張）
- **クライアントAI**: 顧客特有のプレッシャー（「来月の結婚式までに痩せたい」「友達がやってるからBIG3やりたい」等）
- ディベートフェーズ: ユーザーと対立AI＋クライアントAIのテキストベースの議論（Phase 1 MVP）

### 固定シナリオ 5本（クライアント・アーキタイプ）

シナリオは「競技種目」ではなく「顧客の目的×属性」で設計する。

| # | アーキタイプ | 目的 | 属性 | 破綻部位 | 典型的な破綻パターン |
|---|------------|------|------|---------|-------------------|
| 1 | デスクワーカー男性（35歳） | ダイエット | 座りっぱなし、腸腰筋硬い、BMI 28 | **腰部** | 腸腰筋が硬い状態でデッドリフト→腰椎過伸展→ヘルニア |
| 2 | 産後ママ（32歳） | 体型戻し＋姿勢改善 | 骨盤底筋弱化、腹直筋離開リスク | **腰部** | 腹圧が保てない状態での高負荷→腰痛悪化 |
| 3 | 中年男性（50歳） | 筋肥大（若返り） | 変形性膝関節症リスク、肥満傾向 | **膝関節** | 体重過多×フルスクワット×過負荷→半月板損傷 |
| 4 | 猫背OL（28歳） | 肩こり解消＋姿勢改善 | 上位交差症候群、胸椎後弯 | **肩関節** | 肩甲骨の可動域不足でオーバーヘッドプレス→インピンジメント |
| 5 | 学生アスリート（20歳） | 競技力向上 | 高頻度練習、睡眠不足、栄養偏り | **膝関節** | オーバートレーニング×睡眠不足→ACL断裂リスク |

**怪我の焦点（3部位に限定）:**
- **腰部（腰痛・ヘルニア）**: デスクワーク→腸腰筋硬化→デッドリフトでの破綻
- **膝関節（膝痛・半月板）**: 肥満・過負荷→スクワット/ジャンプでの破綻
- **肩関節（インピンジメント）**: 猫背→オーバーヘッド種目での破綻

### Step 2: 残酷な結果の可視化（Consequence）

- V7推論エンジン（V7-Risk + V7-Adaptation）が処方メニューに基づく結果を確定的に計算
- **レーダーチャート**: パフォーマンス指標（筋力、持久力、柔軟性、安定性、回復度等）の変化を可視化
- **バイオ・バタフライエフェクト**: 小さな判断ミスが連鎖的に大きな結果を生む過程を表示
- **失敗時**: 怪我の発生（Phase 1: テキスト+レーダーチャート表示、Phase 2: 3Dアバター表示）
- **ゲームオーバー条件**: ACL断裂等の重大怪我→選手のシーズン終了→シミュレーション終了

### Step 3: AIメンターによる種明かし（AAR — After Action Review）

- ベテランATCペルソナのAIメンター（Geminiテキスト生成）
- **cause_breakdown**: V7推論エンジンの計算結果を論理的に分解して解説
  - 「あなたの処方 X が生体力学的に Y を引き起こし、結果として Z のリスクが N% 上昇した」
  - エビデンスレベル表示（論文参照）
- フィードバックは教育的・建設的（責めるのではなく、なぜそうなったかを説明）

### Step 4: タイムトラベルと再挑戦（Retry & What-If）

- **タイムトラベルスライダー**: 任意の時点に巻き戻し可能
- **修正エクササイズDB**: 最適解に近いメニューの候補を提示（RAG）
- **What-Ifシミュレーション**: 修正メニューでV7推論エンジンを再実行し、結果を比較表示
- 比較レーダーチャート: 元の結果 vs 修正後の結果をオーバーレイ表示

### 4体のAIエージェント

- 出力トークン上限: 各ターン最大500トークン（コスト保護）
- ユーザー別レートリミット: 1分あたり10リクエスト

## 6. 必須コア要件

### 6-1. DBスキーマ要件

```sql
-- ユーザー管理
users (id, email, role, plan, created_at, updated_at)
user_profiles (id, user_id, display_name, affiliation, experience_years, specialty)

-- シナリオ管理
scenarios (id, title, description, difficulty, athlete_profile_json, initial_condition_json, created_by, is_template, created_at)
scenario_sessions (id, user_id, scenario_id, role_choice, status, started_at, completed_at)

-- 意思決定ログ
session_turns (id, session_id, turn_number, user_action_json, ai_responses_json, v7_input_json, v7_output_json, radar_data_json, created_at)

-- エクササイズDB（既存メニューDBの参照テーブル）
exercises (id, name, category, target_muscles, intensity_level, contraindications, evidence_level, source_reference)
prescribed_menus (id, session_id, turn_number, exercises_json, rationale_text)

-- AAR・フィードバック
aar_records (id, session_id, cause_breakdown_json, mentor_feedback_text, evidence_references, created_at)

-- タイムトラベル
timeline_snapshots (id, session_id, turn_number, snapshot_json, created_at)
whatif_comparisons (id, session_id, original_turn, modified_menu_json, modified_v7_output_json, comparison_data_json, created_at)

-- 学習進捗
user_progress (id, user_id, total_sessions, successful_outcomes, failed_outcomes, avg_risk_score, skill_radar_json, updated_at)

-- プラン・課金（Stripe連携）
subscriptions (id, user_id, stripe_subscription_id, plan, status, current_period_start, current_period_end)
usage_tracking (id, user_id, month, scenarios_used, api_calls, llm_tokens_used)
```

**インデックス要件:**
- `scenario_sessions(user_id, status)` — ユーザーの進行中セッション取得
- `session_turns(session_id, turn_number)` — セッション内のターン順序
- `user_progress(user_id)` — ユーザー進捗取得
- `usage_tracking(user_id, month)` — 月間利用量チェック

**RLS（Row Level Security）:**
- 全テーブルにRLS有効化
- ユーザーは自分のデータのみ読み書き可能
- Education/B2Bプラン: 管理者は所属メンバーのデータを読み取り可能
- シナリオテンプレート（is_template=true）は全ユーザーが読み取り可能

### 6-2. 認証・アクセス制御要件

| 項目 | 仕様 |
|------|------|
| 認証方式 | Better Auth（OSS、メール/パスワード）。Neon PostgreSQLにテーブル直接作成。Phase 2でSSO追加検討 |
| RBAC | student, pro_user, educator, team_admin, system_admin |
| セッション | JWT（Better Auth発行 → Honoミドルウェアで検証） |
| アクセス制御 | Drizzle ORM + Honoミドルウェアで認証チェック（Supabase RLSの代替） |
| レートリミット | Free: 月3シナリオ / LLM 10req/min、クレジット購入済: クレジット残高内で無制限 / LLM 30req/min |

### 6-3. デザイン要件

**全体方針:**
- ダークモード基調（長時間学習の目の負担軽減）
- 医療・スポーツ科学の信頼感を演出（クリーン、プロフェッショナル）
- 「ゲーム感」は控えめ、「シミュレータ」としての専門性を前面に

**主要画面構成（Phase 1 MVP）:**

| 画面 | 概要 |
|------|------|
| ダッシュボード | 学習進捗、過去のセッション一覧、スキルレーダーチャート |
| シナリオ選択 | 難易度別シナリオ一覧、仮想アスリートプロフィールプレビュー |
| シミュレーション画面（メイン） | 左: アスリート情報+レーダーチャート、中央: ディベートチャット、右: メニュー処方パネル |
| AAR画面 | cause_breakdown表示、比較レーダーチャート、エビデンスリンク |
| タイムトラベル画面 | タイムラインスライダー、What-If比較ビュー |
| プロフィール/設定 | アカウント情報、学習統計、プラン管理 |

**レーダーチャート仕様:**
- 軸: 筋力(Strength)、持久力(Endurance)、柔軟性(Flexibility)、安定性(Stability)、回復度(Recovery)、メンタル(Mental)
- カラー: 現在値（プライマリカラー）、目標値（グレー破線）、危険閾値（赤破線）
- アニメーション: Framer Motionで値変化をスムーズに遷移

**タイムトラベルスライダー仕様:**
- 水平タイムライン、各ターンをノードとして表示
- ノードの色: 緑（良い結果）、黄（警告）、赤（怪我発生）
- ドラッグで任意の時点に移動、その時点のレーダーチャートが即座に更新

## 8. 商用AI 4大防壁の適用

| 防壁 | DILEMMAでの具体適用 |
|------|-------------------|
| 【防壁1】モック排除 | 全シナリオは実際のV7推論エンジンで計算。ダミースコア禁止 |
| 【防壁2】AIセキュリティ | 4体のLLMエージェントに個別のシステムプロンプト+入力サニタイズ+出力ガードレール |
| 【防壁3】コスト保護 | ユーザー別LLMトークン追跡（usage_tracking）、プラン別レートリミット |
| 【防壁4】耐障害性 | V7推論失敗時: Go→TSフォールバック。LLM応答失敗時: 最大3回指数バックオフリトライ |

## 9. 教育的価値とMoat

### データフライホイール（Phase 3）

```
学生の意思決定ログ
  → 匿名集計
  → 「典型的な判断ミスパターン」の抽出
  → プロ向けPace-platformの教師データ
  → V7推論エンジンの精度向上
  → より良いシミュレーション結果
  → 学生の学習体験向上
  → さらに多くの意思決定ログ
```

### 競合優位性

- V7推論エンジンの非線形生体力学推論は模倣困難（pace-inferenceの8ms/6.1MB）
- 「確定的推論 + LLMナラティブ」の分離アーキテクチャが教育信頼性を担保
- 既存Pace-platformのエクササイズDBがRAGナレッジベースとして機能
